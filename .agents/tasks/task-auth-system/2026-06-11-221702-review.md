# JWT cookie auth with group-based RBAC for Maestro

Maestro gains a full authentication and authorization layer: username/password login backed by bcrypt-hashed credentials, JWT session tokens stored in httponly cookies, and a five-group RBAC model (Administrators, Viewers, Approver, Operators, Developers). The implementation spans models, migration with seed data, a service/repository layer, FastAPI dependencies for permission gating, a login page, a user management CRUD page, and route protection across all existing UI/API endpoints.

**Watch for:** The session cookie lacks the `secure` flag, meaning tokens transmit in cleartext over HTTP (confirmed). The `SECRET_KEY` default of `"change-me-in-production"` ships functional — there is no startup guard or warning when the default is active (confirmed). The `not_authenticated_handler` unconditionally returns a 302 redirect even for non-HTML API consumers (confirmed). The N+1 query pattern in the user listing partial fetches groups per-user in a loop (confirmed).

## High-level view

The auth architecture follows the project's existing layered pattern: models in `database/models.py`, a repository for persistence, a service for business logic (hashing, JWT encode/decode), and FastAPI `Depends()` functions that wire permission checks into routes. Permission levels are defined as module-level callables (`can_view`, `can_approve`, `can_operate`, `can_admin`). The five groups form a rough hierarchy enforced by listing allowed groups per permission level rather than encoding formal inheritance.

The session token is a stateless JWT with user ID and 24-hour expiry, set as an httponly cookie. There is no token revocation mechanism — deactivating a user is checked on each request by re-fetching from the database, which provides soft revocation at the cost of a DB query per request. The cookie is not marked `secure`, so the token will be sent over plain HTTP connections.

The migration seeds an admin user with a known password (`chang3m3`) and hardcodes group IDs 1-5. The hardcoded password means every deployment starts with the same credential, and there is no forced rotation on first login.

Test coverage spans 20 unit tests for the service/repo/dependency layer and 31 route tests covering all four permission levels, but relies entirely on dependency overrides rather than real JWT cookie flows through the middleware stack.

## Session cookie security posture

The login route sets the session cookie with `httponly=True` and `samesite="lax"` but omits `secure`:

```python
response.set_cookie(
    key="maestro_session",
    value=token,
    httponly=True,
    samesite="lax",
    max_age=86400,
)
```

Without `secure=True`, the cookie is sent on any HTTP request, making the JWT interceptable on non-TLS connections. In a production environment behind TLS termination the proxy handles encryption, but the cookie itself offers no protection if any hop is plain HTTP. Fix: `secure=(settings.environment != "local")`.

The `SECRET_KEY` setting defaults to `"change-me-in-production"` with no runtime validation. If a deployment forgets to set the environment variable, all JWTs are signed with a publicly-known key — effectively fail-open auth.

## NotAuthenticatedException handler asymmetry

In `main.py`:

```python
@app.exception_handler(NotAuthenticatedException)
async def not_authenticated_handler(request: Request, exc: NotAuthenticatedException):
    accept = request.headers.get("accept", "")
    if "text/html" in accept:
        return RedirectResponse(url="/ui/login", status_code=302)
    return RedirectResponse(url="/ui/login", status_code=302)
```

Both branches do the same thing. A non-browser client hitting a protected endpoint gets a 302 to an HTML login page instead of a 401 JSON response. The non-HTML branch should return `JSONResponse(status_code=401, content={"detail": "Not authenticated"})`.

## N+1 in user listing and per-request query cost

The `users_partial_list` handler fetches all users, then loops through them calling `get_user_groups` for each one individually. With 50 users this is 51 queries. A single query joining through `UserGroupAssociation` to `Group` would collapse this.

Additionally, routes gated with `can_approve`/`can_operate`/`can_admin` issue two DB queries per request: one in `get_current_user` (fetch user by ID) and another in the group-check dependency (fetch user's groups). This is the cost of stateless JWT + real-time deactivation checks. Worth noting as the deployment scales, though acceptable for an internal tool.

## Password and user lifecycle gaps

The `create_user` route validates only non-emptiness for username and password — no minimum length. The change-password route enforces a 4-character minimum. This asymmetry means an admin could create a user with a single-character password.

The `toggle-active` endpoint has no self-protection: an administrator can deactivate themselves or the last remaining administrator. Once the sole admin is inactive, no one can re-activate them through the UI (the endpoint requires `can_admin`). A check like "cannot deactivate if this is the only active administrator" would prevent lockout.

## Test coverage gaps

Not tested: the `NotAuthenticatedException` handler behavior for non-HTML Accept headers (both branches currently do the same thing anyway); the actual cookie-based flow end-to-end (set cookie on login, use it on subsequent request without dependency overrides); group escalation edge cases (user belonging to multiple groups with conflicting permissions).
