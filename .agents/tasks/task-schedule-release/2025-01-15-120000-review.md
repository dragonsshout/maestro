# Scheduled release execution with background dispatch loop

This change adds a full scheduling subsystem to Maestro: users can schedule a release for future execution via REST API or the web UI, and a 30-second polling loop dispatches due schedules by calling the existing `execute_release` path. The implementation follows established patterns — the background task mirrors `timeout_checker`, the repository mirrors `ExecutionRepository`, and the UI integrates via HTMX partials.

**Watch for:** The `OrchestratorService.__new__` bypass in the scheduler task skips `__init__` and manually wires attributes, creating a fragile coupling to internal field names (confirmed). The background tasks spawned via `asyncio.create_task` after `execute_release` run with no error handling and no reference retention, so exceptions are silently lost and the tasks can be garbage-collected (confirmed). There is no row-level locking or status transition guard in `_process_due_schedules`, so concurrent poll cycles (or multiple workers) could double-dispatch a schedule (likely).

## High-level view

The data layer uses a composite index on `(name, status)` for the duplicate-pending check, proper foreign keys to both `orchestrator_descriptor` and `release_execution`, and an Alembic migration that chains off the latest revision.

The service validation has a subtle gap: there's no minimum lead-time, so a schedule 1 second in the future will be accepted and may expire before the next 30-second poll cycle. Probably fine operationally but worth knowing.

The background dispatch loop (`_process_due_schedules`) is the riskiest part. It constructs `OrchestratorService` by bypassing its constructor with `__new__`, manually assigning three fields. If `OrchestratorService.__init__` ever gains a new dependency or renames a field, the scheduler silently operates on an incomplete object. After `execute_release` queues background tasks into a `BackgroundTasks()` instance, the code iterates them and fires-and-forgets via `asyncio.create_task` without saving references or awaiting results. Those tasks (the `process_workflow` call) can vanish on garbage collection or swallow exceptions.

The UI cancel route swallows `ValueError` silently, which means the user sees a refreshed list with no indication that cancellation failed. The `datetime-local` HTML input sends a timezone-naive string; the UI route patches in UTC, which is correct only if the user intends UTC — there's no timezone indicator in the form.

<details>
<summary>Issues (5)</summary>

1. **`__new__` bypass for OrchestratorService** — The scheduler constructs the service by skipping `__init__` and hand-assigning fields. Any change to `OrchestratorService.__init__` (new field, rename) silently breaks the scheduler at runtime. Use a factory method or pass dependencies explicitly through a constructor overload instead.
2. **Fire-and-forget background tasks with no reference** — `asyncio.create_task(task.func(*task.args, **task.kwargs))` discards the task reference. Exceptions are silently dropped and the task object may be garbage-collected before completion. Store references and add exception logging.
3. **No concurrency guard on schedule dispatch** — `_process_due_schedules` fetches pending rows without `SELECT ... FOR UPDATE`. If two workers (or overlapping poll cycles due to slow execution) run simultaneously, both can fetch the same schedule and call `execute_release` twice. Add `with_for_update(skip_locked=True)` or a status transition CAS.
4. **UI cancel silently swallows errors** — The `cancel_schedule_ui` route catches `ValueError` and does nothing, returning the list as-is. The user has no feedback that cancellation failed.
5. **Timezone assumption in UI scheduling form** — The `datetime-local` input has no timezone indicator. The backend assumes UTC, but users in other timezones will schedule at the wrong time. Display "UTC" next to the input or convert from the user's locale.

## Service construction via `__new__` in the dispatch loop

The scheduler builds its `OrchestratorService` instance like this:

```python
svc = OrchestratorService.__new__(OrchestratorService)
svc.repository = orch_repo
svc.execution_repo = exec_repo
svc.jenkins_service = jenkins_service
```

This bypasses `__init__` entirely and manually wires three fields by name. If `OrchestratorService.__init__` gains a fourth dependency (say, a `GithubService`), the scheduler will operate on an instance with a missing attribute and crash only when that attribute is accessed — possibly deep in the `execute_release` path after the schedule has already been marked as in-progress. The `timeout_checker` avoids this problem because it doesn't instantiate `OrchestratorService` at all; it works directly with repos and models.

A safer pattern: add a classmethod like `OrchestratorService.from_session(session)` that constructs the full object with all deps, or accept the individual repos as arguments to a static `execute_release_from_schedule` method.

## Fire-and-forget `asyncio.create_task` with no reference

After `execute_release` returns, the scheduler iterates `bg.tasks` and spawns them:

```python
for task in bg.tasks:
    asyncio.create_task(task.func(*task.args, **task.kwargs))
```

Two problems:

1. The returned `Task` object is not stored anywhere. CPython's event loop holds only a weak reference to tasks. If garbage collection runs before the task completes, the task can be cancelled silently. The standard mitigation is to store references in a set (the "background tasks" pattern from the asyncio docs).

2. If `task.func` raises, the exception is never observed. It becomes an "unhandled exception in task" warning in logs (Python 3.11+), but the schedule remains marked as "executed" because `mark_executed` was called before the background tasks ran.

## No row-level locking between check-and-dispatch

`get_pending_due_schedules` does a plain SELECT. Between reading the rows and calling `mark_executed`/`mark_failed`, there's no database lock preventing another concurrent process from reading the same rows. In a single-worker deployment this is unlikely to manifest because the 30-second sleep provides natural serialization, but if Maestro is ever scaled horizontally (multiple Uvicorn workers, multiple containers), duplicate dispatches become possible. The timeout_checker has the same structural weakness but is less dangerous because marking a step as timed-out is idempotent; triggering `execute_release` twice for the same schedule is not — the second call will hit the "active execution" guard and mark the schedule as failed, a confusing outcome for the user.

Adding `with_for_update(skip_locked=True)` to the pending-schedules query would let concurrent workers safely partition work.

## UI cancel swallows errors silently

```python
@router.delete("/schedule/{schedule_id}", response_class=HTMLResponse)
async def cancel_schedule_ui(...):
    try:
        await scheduler_service.cancel_schedule(schedule_id)
    except ValueError:
        pass  # <-- user gets no feedback
    schedules = await scheduler_service.get_all_schedules()
    return templates.TemplateResponse(...)
```

If the schedule was already cancelled or executed by the time the user clicks "Cancel", the `ValueError` is swallowed and the list refreshes with no error banner. Return an error partial or include an error flag in the template context.

## Test coverage gaps

Not tested: the fire-and-forget `asyncio.create_task` loop (the test mocks it away), the timezone-naive-to-UTC conversion in the UI route, and the behavior when `mark_executed` or `mark_failed` encounters a missing row (they silently no-op via `if schedule:` guards).
